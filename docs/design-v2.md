# Adaptive UI — Design Specification

## 1. Product

The app has two visible zones:

- **Top adaptive surface**: replaced each turn with a new `UiDoc` generated by
  a UI subagent. The surface renders interactive components: text, code, radio
  groups, button rows, multiline text inputs, math text, and chat transcripts.
- **Bottom input bar**: a persistent `SynEdit` text field. The user types here
  and submits with Ctrl+Enter or Cmd+Enter.

The adaptive surface is not a static chat log. Each turn the UI subagent
produces a complete UI document that replaces the previous one. This gives the
subagent full control over what the user sees: a quiz question, an essay
prompt, study notes, a decision prompt, or a transcript.

Supported interaction patterns:

| Pattern | Components used | Flow |
|---|---|---|
| Quiz | radio + buttons | One question per turn. Submit answer, get feedback, next question, final score. |
| Essay | textInput + buttons | Prompt appears, user writes answer, submits, gets graded feedback. |
| Yes/No decision | buttons (yes/no) or radio | User picks a choice, next state depends on the answer. |
| Study notes | text, code, math | Read-only content. No submit required. |
| Chat transcript | transcript | Read-only conversation history. |
| Adaptive task | any combination | General mode. The subagent picks the layout and components. |

## 2. Hard Constraints

- Application code is Nim only.
- Build with `-d:sdl3`. Use `uirelays` for the window, event loop, and
  drawing surface.
- Use `uirelays/layout.parseLayout` for all layout rectangles. Do not write a
  replacement layout solver.
- Use `SynEdit` for every text-like surface: labels, code views, transcript
  views, and multiline text inputs.
- Use `jsonx` for config, state, and agent response objects. Do not use
  `std/json` for project data models.
- Use `relay` and `openai/chat` for model requests. Never call blocking
  `makeRequest` from the render loop.
- Keep modules procedural. Use plain objects and procs. Avoid `method` unless
  runtime dispatch is proven necessary.
- Keep state explicit. The main loop owns `AppState` and passes it by `var`.

## 3. Source Layout

```
src/
  adaptive_ui.nim            # public re-export
  adaptive_ui/
    app.nim                  # window, event loop, frame orchestration
    config.nim               # typed app config via jsonx
    agent.nim                # OpenAI/Relay request construction and polling
    ui_doc.nim               # UiDoc, UiArea, UiKind, UiEvent types
    ui_parse.nim             # parse UiDoc JSON, validate, fallback
    ui_render.nim            # resolve layout, render components, return events
    components.nim           # ComponentState, UiRuntime, event constructors
    interaction.nim          # convert UI events into chat-agent text
    learning.nim             # local scripted quiz state
    live_flow.nim            # live flow commands, prompts, and intro documents
    transcript.nim           # format chat history into transcript UiDoc
    debug_log.nim            # bounded raw-response diagnostics
    markdown_view.nim        # markdown-to-readable-text preprocessor
    math_view.nim            # LaTeX-like text substitution
    skill_files.nim          # load and summarize SKILL.md files
```

## 4. Data Flow

One user turn follows this path:

1. User types in the bottom input and presses Ctrl+Enter.
2. `app.nim` reads the text from the input `SynEdit`, clears it, and passes
   the text to `handleSubmittedInput`.
3. If the input starts with `/`, it is parsed as a command (`/adaptive`,
   `/chat`, `/quiz`, `/essay`, `/debug`). Otherwise it is submitted to the
   current flow.
4. For live flows, `agent.nim` appends a user message to the conversation
   history and enqueues a chat request through Relay.
5. Each frame, `pollLiveAgent` calls `pollAgent` to check for completed
   requests.
6. When a chat response arrives, it is appended to the history and a UI
   request is immediately enqueued. The UI request includes the full
   conversation history, the current UiDoc, and a flow-specific hint.
7. When the UI response arrives, it is parsed with `jsonx` into a `UiDoc`.
   If parsing or validation fails, the previous UiDoc is kept and an error
   is shown in the status bar.
8. `ui_render.nim` resolves `doc.layout` with `parseLayout` and `resolve`,
   offsets the rects into the adaptive parent area, and renders each area.
9. Components draw inside their named cells and may emit a `UiEvent` on user
   interaction (click, selection, text submit).
10. `interaction.nim` converts `UiEvent`s plus current UI values into a text
    summary that becomes the next user message.

Chat and UI requests are sequential: chat first, then UI. The app never has
more than one pending request of each kind at a time.

## 5. UiDoc Contract

The UI subagent returns a single JSON object. The object contains markdown
text fields inside a typed envelope. This gives deterministic parsing while
using markdown as the authoring format for text content.

### Types

```nim
type
  UiKind* = enum
    ukText, ukCode, ukRadio, ukButtons, ukTextInput, ukMath, ukTranscript

  UiOption* = object
    id*: string
    label*: string
    selected*: bool

  UiArea* = object
    name*: string
    kind*: UiKind
    text*: string
    id*: string
    options*: seq[UiOption]
    language*: string
    placeholder*: string
    submitLabel*: string

  UiDoc* = object
    version*: int
    title*: string
    layout*: string
    areas*: seq[UiArea]
    focus*: string
```

### JSON kind names

The `kind` field in JSON uses these exact lowercase strings:

| JSON string | Nim enum | Renderer |
|---|---|---|
| `text` | `ukText` | read-only `SynEdit` |
| `code` | `ukCode` | read-only `SynEdit` with language and line numbers |
| `radio` | `ukRadio` | direct-drawn radio group with hit testing |
| `buttons` | `ukButtons` | direct-drawn button row with hit testing |
| `textInput` | `ukTextInput` | editable `SynEdit` with optional submit button |
| `math` | `ukMath` | text view with LaTeX-to-ASCII substitutions |
| `transcript` | `ukTranscript` | read-only `SynEdit` with console language |

### Validation rules

- `version` must be `1`.
- `layout` must be a non-empty string that `parseLayout` accepts.
- Every `area.name` must appear as a cell in `layout`.
- Area names must be unique.
- `radio` and `buttons` areas must have a non-empty `id` and at least one
  option with non-empty `id` and `label`.
- `textInput` areas must have a non-empty `id`.
- `focus`, if non-empty, must name a cell in `layout`.
- Unknown kind strings cause a parse error.

### Events

```nim
type
  UiEventKind* = enum
    ueNone, ueClick, ueSelect, ueSubmitText

  UiEvent* = object
    kind*: UiEventKind
    area*: string
    id*: string
    value*: string
```

| Event kind | When | Fields |
|---|---|---|
| `ueSelect` | Radio option clicked | `id` = control id, `value` = option id |
| `ueClick` | Button clicked | `id` = button option id |
| `ueSubmitText` | Text input submitted | `id` = control id, `value` = text content |

### Fallback document

If parsing or layout fails, the renderer must substitute this document:

```json
{
  "version": 1,
  "title": "Adaptive UI",
  "layout": "| main, * |",
  "areas": [{ "name": "main", "kind": "text", "text": "The generated UI could not be rendered." }]
}
```

## 6. Component Rendering

All rendering is immediate-mode. No retained widget tree. State that must
survive between frames is stored in `UiRuntime`:

- `components`: a table of `ComponentState` keyed by component id (or area
  name if id is empty). Each entry holds an optional `SynEdit`, the last
  rendered text, language, kind, and the current selection.
- `focus`: the name of the focused area.
- `status`: a short status message.

### Rendering rules

- **text, transcript, code, math**: synced to a read-only `SynEdit` via
  `setLabel`. The editor is reused across frames as long as the component key
  stays the same. Language and text are updated only when they change.
- **radio**: direct drawing with hit testing. Click sets the selected option in
  `ComponentState.selectedOption` and emits `ueSelect`.
- **buttons**: direct drawing with hit testing. Click emits `ueClick`.
- **textInput**: editable `SynEdit`. Content persists in the editor buffer keyed
  by component id. Submit on Ctrl+Enter or by clicking a submit button drawn
  inside the same area (when `submitLabel` is set). Emits `ueSubmitText`.

### Focus routing

- The renderer sets focus to `doc.focus` on the first frame if no focus exists.
- Mouse clicks inside a layout cell set focus to that cell's area name.
- Only the focused area receives keyboard events.

## 7. Layout

The app uses two layout levels:

**Outer layout** (hardcoded in `app.nim`):

```
| adaptive, * |
| input, 4 lines |
| status, 1 line |
```

**Inner layout** (from `UiDoc.layout`, provided by the UI subagent):

Any valid `uirelays` markdown table. The inner rects are resolved using the
adaptive area's width and height, then offset by the adaptive area's screen
origin.

```nim
let cells = parseLayout(doc.layout).resolve(area.w, area.h, lineHeight, gap = 2)
for name, r in cells.mpairs:
  r = rect(area.x + r.x, area.y + r.y, r.w, r.h)
```

## 8. Agent Roles

### Chat agent

- Receives the user's text (or a summary of UI interactions).
- Maintains the conversation history and task state.
- Produces plain text responses that include enough structured detail for the
  UI subagent to choose appropriate components.
- Grades quiz answers and essay submissions when relevant.

System prompt: `ChatSystemPrompt` in `agent.nim`, extended with skill file
summaries when available.

### UI subagent

- Receives the full conversation history, the current UiDoc JSON, and a
  flow-specific hint.
- Returns a single UiDoc JSON object.
- Never produces free-form text, markdown fences, or commentary.
- Uses only the supported component kinds listed in section 5.

System prompt: loaded from `prompts/ui-subagent-system.md`, or
`DefaultUiSystemPrompt` in `agent.nim` if the file is missing.

Structured output: the UI request uses `formatJsonSchema` with a strict JSON
Schema that matches the UiDoc shape. This constrains the model output.

### Flow hints

Each live flow sends a hint to the UI subagent:

| Flow | Hint |
|---|---|
| Adaptive | Choose the smallest supported UI that fits the task. Do not default to quiz or essay. |
| Chat | Prefer transcript or text. |
| Quiz | One radio area for choices, one buttons area for submit/next/finish. Keep option ids stable. |
| Essay | One prompt area, one textInput area, one buttons area for submit. |

## 9. Live Flow Commands

Commands typed in the bottom input bar:

| Command | Action |
|---|---|
| `/adaptive [text]` | Switch to adaptive mode, submit optional text. |
| `/chat [text]` | Switch to chat mode, submit optional text. |
| `/quiz [text]` | Switch to quiz mode, submit optional text. |
| `/essay [text]` | Switch to essay mode, submit optional text. |
| `/debug` | Show the debug log as a UiDoc. |
| anything else | Submit to the current flow. |

## 10. Error Handling

- **UI response fails JSON parsing**: keep the previous UiDoc, show error in
  status, store raw text in `DebugLog`.
- **`parseLayout` raises**: replace the document with the fallback (single
  text area).
- **Unsupported area kind**: rejected at parse time; does not reach the
  renderer.
- **Network error**: shown in status. Previous UiDoc is kept. App stays
  running.
- **Missing API key**: app opens in preview mode showing the intro screen.
  Status says to set `OPENAI_API_KEY`.

## 11. Config

Loaded from `adaptive_ui.json` (or path passed to `runAdaptiveApp`):

```json
{
  "apiUrl": "https://api.openai.com/v1/chat/completions",
  "apiKeyEnv": "OPENAI_API_KEY",
  "chatModel": "gpt-4.1-mini",
  "uiModel": "gpt-4.1-mini",
  "timeoutMs": 30000,
  "skillRoots": ["/home/user/.agents/skills", "/home/user/.codex/skills"]
}
```

If the file does not exist, defaults are used. If the API key environment
variable is empty, the app runs in preview mode.

## 12. Testing

- Tests are standalone Nim files using `doAssert`.
- No `unittest` dependency.
- Test pure parsing, state transitions, event construction, and flow logic.
- No test requires network access or API keys.
- UI rendering is verified via compile-only smoke builds.

Run all tests:

```sh
nim c -d:sdl3 -r tests/tester.nim
```

## 13. Style

- Indent with 2 spaces.
- `PascalCase` for types, `camelCase` for procs, vars, and fields.
- Prefer plain `object` passed by `var`. Prefer top-level procs.
- Use `proc` by default. Use `func` only for pure queries.
- Export only what other modules need.
- Do not hide app state in globals. Backend relay globals from `uirelays` are
  acceptable because that is how the library works.

## 14. Build

The repo includes `config.nims` that sets `-d:sdl3` by default, so either of
these works:

```sh
nim c -r tests/tester.nim
nim c -d:sdl3 -r tests/tester.nim
```

To run the adaptive app:

```sh
nim c -r examples/adaptive_app.nim
./examples/adaptive_app
```

## 15. Open Work Items

These items need attention in future work sessions:

1. **Streaming**: both chat and UI responses are received in full before
   rendering. Add streaming without changing the UiDoc contract.

2. **Session persistence**: conversation history is lost when the app closes.
   Add save/load to the agent module.

3. **Editor state cleanup**: when the UI subagent generates different
   component ids across turns, old `ComponentState` entries remain in the
   table. Add a cleanup step that removes entries not present in the current
   UiDoc.

4. **Flow state reset**: switching from one live flow to another must reset
   `UiRuntime` so that stale selection state from the previous flow does not
   persist.

5. **Automated render tests**: add pixel-level or snapshot-based UI tests
   beyond compile-only smoke builds.
