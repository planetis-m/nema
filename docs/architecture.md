# Adaptive UI Architecture

This document defines the first implementation target for the adaptive UI app. It is intentionally small. The goal is to ship one reliable path before adding richer UI generation.

## Product Shape

The app has two visible zones:

- Top adaptive surface: generated by a UI subagent and rendered as interactive components.
- Bottom input bar: normal agent text input, always available.

The adaptive surface can render:

- normal chat or study notes
- one quiz question with radio choices
- one essay prompt with a multiline answer box
- simple yes/no decision prompts
- score or feedback summaries
- read-only code blocks with syntax highlighting
- basic math text

The app does not need to implement every learning/game flow directly. It only needs a small rendering contract that lets the agent drive those flows one turn at a time.

## Hard Constraints

- Nim only for application code.
- Use `uirelays` for UI and build with `-d:sdl3`.
- Use `uirelays/layout.parseLayout` for all layout rectangles.
- Use `SynEdit` for editable text, labels, code views, and multiline answer areas.
- Use `jsonx` for typed configuration and agent response parsing.
- Use `relay` and `openai` for model calls.
- Do not use `std/json` for configuration, state, or agent response parsing.
- Keep modules procedural. Prefer plain objects and procs over methods.
- Keep state explicit. The main loop owns `AppState` and passes it by `var`.

## High-Level Modules

Target source layout:

```text
src/
  adaptive_ui.nim
  adaptive_ui/
    app.nim
    config.nim
    agent.nim
    ui_doc.nim
    ui_parse.nim
    ui_render.nim
    components.nim
    markdown_view.nim
    skill_files.nim
    math_view.nim
tests/
  test_ui_doc_parse.nim
  test_control_events.nim
  test_skill_files.nim
examples/
  learning_demo.nim
```

Module responsibilities:

- `adaptive_ui.nim`: public re-export for app modules that should be stable.
- `app.nim`: window setup, event loop, frame orchestration, focus routing.
- `config.nim`: typed app config loaded and saved with `jsonx`.
- `agent.nim`: OpenAI/Relay request construction, enqueue, poll, parse.
- `ui_doc.nim`: typed UI document and interaction event types.
- `ui_parse.nim`: parse UI agent output into `UiDoc`.
- `ui_render.nim`: resolve layout and call component renderers.
- `components.nim`: immediate-mode controls, most backed by `SynEdit`.
- `markdown_view.nim`: tiny markdown display support for the first prototype.
- `skill_files.nim`: locate, load, and summarize `SKILL.md` files.
- `math_view.nim`: basic inline and block math rendering fallback.

## Data Flow

One user turn follows this path:

1. User edits the bottom input and submits.
2. `agent.nim` appends a user message to `Conversation`.
3. Chat agent request is enqueued through Relay.
4. Each frame polls Relay for completed requests.
5. When the chat response arrives, append it to `Conversation`.
6. Enqueue a UI agent request with the latest conversation and current `UiDoc`.
7. When the UI response arrives, parse it with `jsonx` into `UiDoc`.
8. `ui_render.nim` resolves `UiDoc.layout` with `uirelays/layout`.
9. Components draw inside their named cells and emit `UiEvent`s.
10. `agent.nim` converts `UiEvent`s into the next user/tool message.

The UI agent decides what to show. The app only validates and renders a small supported component set.

## UiDoc Contract

The UI agent returns JSON, not raw free-form markdown. The JSON contains markdown text fields. This gives deterministic parsing while still using markdown as the authoring format.

```nim
type
  UiDoc* = object
    version*: int
    title*: string
    layout*: string
    areas*: seq[UiArea]
    focus*: string

  UiArea* = object
    name*: string
    kind*: UiKind
    text*: string
    id*: string
    options*: seq[UiOption]
    language*: string
    placeholder*: string
    submitLabel*: string

  UiKind* = enum
    ukText,
    ukCode,
    ukRadio,
    ukButtons,
    ukTextInput,
    ukMath,
    ukTranscript

  UiOption* = object
    id*: string
    label*: string
    selected*: bool
```

Rules:

- `layout` must be a markdown table accepted by `parseLayout`.
- Each `UiArea.name` must match a cell name in `layout`.
- Unknown cells are ignored.
- Missing areas render as empty panels.
- `focus` is optional and names the first focused area.
- `text` is markdown-flavored plain text, not HTML.
- `ukCode.language` maps to `SynEdit.lang` where possible.
- `ukTextInput` uses a persistent `SynEdit` buffer keyed by `id`.
- `ukRadio` and `ukButtons` emit `UiEvent` when clicked or activated.

## Component Set

Use one `ComponentState` table keyed by area `id` or area `name`.

Initial components:

- Text view: read-only `SynEdit` via `setLabel`.
- Code view: read-only `SynEdit`, line numbers enabled, language set.
- Radio group: direct drawing plus hit testing. Emits selected option id.
- Buttons: direct drawing plus hit testing. Emits button id.
- Text input: editable `SynEdit`. Emits submit event on Ctrl+Enter or a paired submit button.
- Math view: render as text first, with better glyph layout later.

Do not build a retained widget tree in the first pass. Store only the state that must survive frames:

- text buffers
- radio selection
- scroll position when provided by `SynEdit`
- focus name
- pending events

## Layout Strategy

The app uses two layouts:

- Outer app layout, owned by the app:

```text
| adaptive, * |
| input, 4 lines |
| status, 1 line |
```

- Inner adaptive layout, provided by `UiDoc.layout`.

At render time:

1. Resolve the outer app layout.
2. Draw the bottom input and status.
3. Clip or offset drawing for the `adaptive` rect.
4. Resolve the UI document layout using the adaptive rect dimensions.
5. Offset each inner cell by the adaptive rect origin before rendering.

`uirelays/layout.resolve` returns screen-origin rects. If resolving inside a subrect, offset each result manually.

## Agent Roles

Chat agent:

- Answers the user.
- Maintains task context.
- Grades quiz or essay submissions when needed.
- Produces text that can be shown in normal chat mode.

UI agent:

- Produces only `UiDoc` JSON.
- Converts current task state into supported UI components.
- Keeps the UI small enough to fit the screen.
- Uses existing choices and scores from the chat agent when available.
- Does not invent app capabilities beyond the supported `UiKind` set.

The first implementation can run these roles sequentially. Parallel requests can come later.

## Error Handling

If the UI response fails JSON parsing:

- Keep the previous `UiDoc`.
- Show a compact error in `status`.
- Store the raw failed text in a debug log buffer.

If `parseLayout` raises:

- Replace the layout with a known fallback:

```text
| main, * |
```

- Render the UI response as `ukText` in `main`.

If an area kind is unsupported:

- Render its `text` as `ukText`.
- Add a status warning.

## First Demo Target

The first demo should be a local scripted learning flow before live model calls are required:

- The top area shows a quiz question.
- The user selects one radio option.
- A button submits the answer.
- The next `UiDoc` shows feedback and another question.
- The final `UiDoc` shows score and correct answers.

This proves the adaptive renderer, component events, and state loop without depending on network access.
